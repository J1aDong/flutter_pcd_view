## Context

当前查看器已经支持背景颜色、点大小、网格、坐标轴等显示配置，并且这些配置会经过 `ViewerConfig`、example 设置页、持久化层以及移动端 native renderer 同步生效。现在点颜色仍然缺失独立配置能力，导致在浅色或特定演示背景下只能依赖原始点数据颜色，或者退回默认白色，配置维度不完整。

这次改动需要同时覆盖：
- `ViewerConfig` 的配置模型与默认值；
- example 设置页的交互入口；
- `ViewerConfigNotifier` 的持久化与恢复；
- 点云打包/渲染链对默认点颜色的使用逻辑。

## Goals / Non-Goals

**Goals:**
- 为查看器增加独立的点颜色配置。
- 让点颜色配置能够通过 example 设置页修改并持久化保存。
- 让未携带自定义颜色的点在 Flutter / native 渲染链中按配置颜色显示。
- 保持现有背景色、点大小等设置项的交互结构不被破坏。

**Non-Goals:**
- 不引入按类别、按高度或渐变色等高级着色模式。
- 不改动点云文件解析逻辑本身。
- 不新增复杂的主题系统或预设色盘管理。

## Decisions

### 1. 在 `ViewerConfig` 顶层新增 `pointColor`

**决定：** 复用当前显示配置模式，在 `ViewerConfig` 顶层增加 `pointColor` 字段，而不是放进 `PerformanceConfig` 或新建独立配置对象。

**理由：**
- 点颜色属于基础显示参数，与背景颜色、点大小同级。
- 可以减少改动面，保持调用方心智一致。
- `copyWith`、相等性判断、默认值扩展都更直接。

**备选方案：**
- 放入新的显示配置对象：结构更细，但当前改动过小，不值得引入额外层级。

### 2. 原始点带颜色时优先保留原色，缺省时回落到配置颜色

**决定：** 渲染链对点颜色采用“点数据原色优先、缺省时使用 `pointColor`”的策略。

**理由：**
- 避免破坏已有带颜色点云的显示结果。
- 同时满足当前“默认白点可配置”这个需求。
- 与用户预期一致：配置是默认值，不是强制覆盖全部点颜色。

**备选方案：**
- 无论原始颜色如何全部覆盖为配置色：实现简单，但会损失已有彩色点云信息。

### 3. 设置页在“背景颜色”下方增加“点颜色”入口

**决定：** 保持当前设置页组织方式，在“显示设置”分组中紧跟背景颜色下方新增点颜色设置。

**理由：**
- 用户已经明确指定入口位置。
- 点颜色与背景颜色属于同一视觉调节区，放在一起最自然。
- 改动最小，不需要重排整个设置页结构。

**备选方案：**
- 放在“点云设置”分组：也可行，但与用户指定位置不一致。

## Risks / Trade-offs

- [原始点颜色判定不清晰] → 需要明确当前 `Point3D.color` 的缺省值语义，避免把合法白色误判为“无颜色”。
- [配置同步遗漏] → 同时检查 Flutter packed scene、native config 更新和设置页持久化，避免只改一层导致显示不一致。
- [设置项继续膨胀] → 当前先保持单色配置，后续如需高级着色再单独提案，不在本次扩展范围内解决。
